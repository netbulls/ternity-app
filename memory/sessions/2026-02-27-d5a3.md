## Topic: Command+Enter vs Enter behavior in Command Palette

User asked to verify whether Cmd+Enter and Enter are wired the same way when the command palette (Cmd+K) is open.

### Findings

Reviewed `command-palette.tsx` and `timer-bar.tsx`:

**Command palette (`command-palette.tsx:113-137`):**

- `handleKeyDown` checks `e.key === 'Enter'` and distinguishes only via `e.shiftKey`:
  - **Enter** (plain) ‚Üí `handleSelect` (copy issue to timer description)
  - **Shift+Enter** ‚Üí `handleStartTimer` (start timer with issue)
  - **Cmd+Enter** ‚Üí NOT separately handled. Since `e.metaKey` is not checked, Cmd+Enter falls through to the same `Enter` branch ‚Üí calls `handleSelect` (same as plain Enter)

**Timer bar (`timer-bar.tsx:118-126`):**

- `handleKeyDown` checks `e.key === 'Enter' && !running` ‚Üí calls `handleStart`
- No modifier key checks at all

### Actual Behavior (corrected after deeper analysis)

The user reported that Enter and Cmd+Enter both **start the timer**, and Shift+Enter causes an error. The root cause is **focus cascading**:

1. User presses Enter in the palette ‚Üí `handleSelect` fires ‚Üí dispatches `ternity-palette-select` event ‚Üí `setOpen(false)`
2. Palette closes/unmounts ‚Üí browser returns focus to the timer-bar input (which was focused before palette opened)
3. The Enter key is still physically down ‚Üí a new `keydown` fires on the timer-bar input
4. Timer-bar's `handleKeyDown`: `e.key === 'Enter' && !running` ‚Üí `handleStart()` ‚Üí starts the timer!

So Enter in the palette effectively: copies description + starts timer (unintentionally).

For Shift+Enter: palette's `handleStartTimer` calls `startTimer.mutate()`, then focus cascades to timer-bar, where the Enter key fires `handleStart()` again ‚Äî potentially a double start request, or the timer-bar starts with empty/old description because the palette's `ternity-palette-select` event wasn't dispatched (Shift+Enter path doesn't dispatch it).
