## Entries Day View — Two Bugs Fixed

### Bug 1: Day view showed entries from all days

**Root cause:** `entries.ts` lines 347-404 — the day-grouping phase iterated all segments of matched entries and created day groups for EVERY date those segments touched, without filtering to the requested `from`/`to` range. So if the user requested only March 5, but a matched entry had segments on Feb 17, Feb 23, Mar 4, Mar 5, all four dates would appear in the response.

**Fix:** Added `if (date < fromDate || date > toDate) continue;` at line 376 before creating groups. One-line fix.

### Bug 2: Multiple segments/entries marked as running

**Root cause:** `timer.ts` `stopRunningSegment()` used `.limit(1)` — it only stopped ONE running clocked segment per user. If a race condition or earlier bug created multiple orphaned running segments, they persisted forever. Each subsequent timer start would only stop one, leaving the rest open. These orphaned segments also caused their entries to be pulled into every date-range query (via the running-segment catch-all at entries.ts lines 190-196).

**Fix:** Rewrote `stopRunningSegment()` to find ALL running clocked segments (no `.limit(1)`), loop through them, stop each one, and audit each stop. Returns the last entry ID stopped.

### CRITICAL: `orgMidnightToUTC` was fundamentally broken

**Root cause:** The function used `new Date(refUTC.toLocaleString('en-US', { timeZone: ORG_TIMEZONE }))` to find the UTC offset. But `new Date()` parses locale strings using the **system timezone**. When the system timezone matches the org timezone (both Europe/Warsaw), the offset is always 0 — meaning `orgMidnightToUTC('2026-03-06')` returned UTC midnight (`2026-03-06T00:00:00Z`) instead of Warsaw midnight (`2026-03-05T23:00:00Z`).

This explains why the day view showed no entries: the SQL range was `00:00Z–23:59Z` (UTC day) instead of `23:00Z–22:59Z` (Warsaw day). Week/month views worked because their wider range masked the 1-hour error.

**Fix:** Rewrote using `Intl.DateTimeFormat` with `timeZoneName: 'longOffset'` to reliably extract the UTC offset (e.g., `GMT+01:00`), then compute midnight = `00:00Z - offset`. Fixed in all 3 copies: `entries.ts`, `dashboard.ts`, `stats.ts`.

Build passes clean.
