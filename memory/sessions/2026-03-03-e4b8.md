### Fix: Timer bar description not saving to server

**Problem:** Editing the description in the timer bar while the timer was running appeared to work (entry list updated in real-time) but the change was never persisted to the database. A page refresh would revert to the original description.

**Root cause:** Two interacting bugs:

1. **Optimistic patch defeated the debounce comparison.** `handleDescriptionChange` called `patchEntry()` which updated both the `['entries']` and `['timer']` query caches optimistically. The debounced save effect then compared `descriptionRef.current` against `currentEntry.description` — but `currentEntry` came from the optimistically-patched `['timer']` cache. Since they were always equal, the PATCH API call **never fired**.

2. **Fingerprint sync loop.** The sync fingerprint included the description (`id::description::projectId`). When the optimistic patch changed `currentEntry.description`, the fingerprint changed, triggering the sync block which set `savedDescriptionRef` to the new unsaved value — compounding the problem.

**Fix (3 parts):**

1. **`savedDescriptionRef`** — Track the last server-confirmed description separately from the cache. The debounce compares against this ref, not `currentEntry.description`. Updated on: initial sync, entry change, and `onSuccess` of the save mutation.

2. **Removed description from fingerprint** — Fingerprint is now `id::projectId` only. Description sync happens through the debounced save mechanism instead. This prevents the optimistic cache patch from creating false sync triggers.

3. **Atomic description on stop** (from prior session) — `POST /timer/stop` now accepts optional `{ description }` and saves it in the same DB transaction as the segment stop. Handles the edge case where the user types and immediately hits stop before the debounce fires.
