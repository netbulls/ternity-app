## Topic: Add impersonation to Command Palette (Cmd+K)

### Context

- User wants admin users to be able to impersonate other people directly from the command palette
- Question: how should the menu be structured? Search people first then jump to actions?

### Current State (discovered)

- **Command palette** (`apps/web/src/components/command-palette.tsx`, ~1030 lines) is fully custom (not using shadcn cmdk primitives). Three item types: entries, Jira issues, nav pages. Already admin-aware for nav items.
- **Impersonation system already fully implemented** — provider, backend hook, API header, banner, transition animations. Currently triggered only from the User Management page.
- **User listing** available via `GET /api/users` (admin-only) → `UserOption[]`, consumable via `useUsers()` hook.
- The palette has a `#` prefix convention for Jira-only search mode.

### Decision: Simple flat approach

- User chose simple approach — no nested menus, just search people and Enter to impersonate
- Admin-only: all people logic gated behind `isAdmin` check

### Implementation (command-palette.tsx)

- Added `person` type to `PaletteItem` union
- Added `@` prefix for people-only search mode (like `#` for Jira)
- During text search (2+ chars), a "View as" section appears with up to 3 matched people
- During `@` mode, uncapped people results shown in "People" section
- Enter on a person calls `setImpersonationTarget()` from existing impersonation provider
- Added `PersonResultRow` component with avatar, name, email, and amber "View as" badge
- Amber-colored mode indicator bar when in `@` mode
- Updated placeholder text and footer hints for admin users
- Non-admin users see zero difference — all people logic guarded by `isAdmin`
- TypeScript builds clean

## Topic: Project Assignment — Implementation

### Decisions

- **User chose V5 (Bulk Matrix)** for Project → Users and **V4 (Inline Table)** for User → Projects
- Both live as drill-down views within existing admin pages (not separate routes)
- V5 accessed via "Members" in project dropdown on Projects page
- V4 accessed via "Projects" in user dropdown on User Management page
- Use existing components only — Switch, DataTable, StatCard, DataTableBulkActions, native `<select>` for roles
- No new Select/Combobox component needed

### Implementation completed

#### Shared types (`packages/shared/src/project-members.ts`)

- `ProjectMemberRow` — user as seen from project side
- `UserProjectRow` — project as seen from user side
- Zod schemas: SetProjectMember, BulkSetProjectMembers, BulkRemoveProjectMembers, UpdateMemberRole, SetUserProject

#### API routes (`apps/api/src/routes/admin-project-members.ts`)

- Project→Users: GET/POST members, PATCH/DELETE individual, bulk-assign, bulk-remove
- User→Projects: GET projects, POST assign, DELETE remove
- All admin-gated with `isRealAdmin()` (impersonation-aware)

#### Frontend hooks (`apps/web/src/hooks/use-admin-project-members.ts`)

- Project direction: useProjectMembers, useAssignProjectMember, useRemoveProjectMember, useUpdateProjectMemberRole, useBulkAssign, useBulkRemove
- User direction: useUserProjects, useAssignUserProject, useRemoveUserProject, useUpdateUserProjectRole

#### Project Members drill-down (Projects page)

- Full implementation in `projects.tsx` — drilldownProject state, member stats, search/filter toolbar, DataTable with member columns, bulk assign/remove bar
- Column defs in `project-member-columns.tsx` — checkbox, user (avatar+name+email), role dropdown, role badge, assigned toggle switch
- "Members" action added to project dropdown in `project-columns.tsx`

#### User Projects drill-down (User Management page)

- Full implementation in `user-management.tsx` — drilldownUser state, project stats, search/filter toolbar, DataTable with project columns, bulk assign/remove bar
- Column defs in `user-project-columns.tsx` — checkbox, project (color swatch+name), client, role dropdown, assigned toggle switch
