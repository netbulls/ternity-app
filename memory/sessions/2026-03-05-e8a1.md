## Wall Chart Footer Bar Alignment & Font Size Fix

**Task:** Fix two issues in the wall chart bottom bar ("People out" footer row):

1. Entries not vertically aligned with header columns
2. Font size inconsistent with header

**Root cause:** The footer row's structure didn't mirror the header row's:

- Missing `flex` and `items-center` on the "People out" label cell — used `py-1.5` padding instead of flexbox centering
- Missing `items-center` and sizing on the empty "Days" spacer cell
- Font size was `scaled(8)` with inline `fontWeight: 600` instead of matching header's `scaled(9)` with `font-semibold` class
- Day count cells used fixed `height: 28` instead of matching header's dynamic height (`rowH` for month view, auto for week view)
- Footer cells didn't carry the same weekend/holiday/today background styling as header cells

**Fix:** Made the footer row structure mirror the header row exactly:

- Added `flex items-center` to both sticky columns
- Changed font size from `scaled(8)` to `scaled(9)` and `fontWeight: 600` to `font-semibold` class
- Changed cell heights from fixed `28px` to use `rowH`/auto like header
- Added weekend/holiday/today background classes to match header
- File: `apps/web/src/pages/leave.tsx` lines 557-607

## Default Project per User (Admin-Only)

**Task:** Add ability for admins to assign a "default project" to each user. Organizational label only — no auto-fill behavior. Data cleanup tool.

**Design decisions:**

- Admin-only setting, not user self-service (separate from the existing `defaultProjectId` in user preferences jsonb which is user self-service)
- Proper FK column on users table rather than jsonb metadata
- FK has ON DELETE SET NULL so deleting a project doesn't break users
- Reuses existing `ProjectSelector` popover component (search, grouped by client, recent projects)

**Changes:**

1. **Schema** (`apps/api/src/db/schema.ts`): Added `defaultProjectId: uuid('default_project_id')` to users table
2. **Migration** (`apps/api/drizzle/0023_user_default_project.sql`): ALTER TABLE ADD COLUMN + FK constraint with ON DELETE SET NULL
3. **API** (`apps/api/src/routes/admin-users.ts`):
   - `GET /api/admin/users` now joins projects + clients to return `defaultProjectId`, `defaultProjectName`, `defaultProjectColor`, `defaultProjectClientName`
   - New `PATCH /api/admin/users/:id/default-project` endpoint — validates project exists, sets or clears the FK
4. **Frontend hook** (`apps/web/src/hooks/use-admin-users.ts`):
   - Extended `AdminUser` type with default project fields
   - Added `useUpdateDefaultProject()` mutation hook
5. **UI column** (`apps/web/src/pages/user-management-columns.tsx`):
   - New "Default Project" column between Employment and Last Active
   - Uses inline `ProjectSelector` component with borderless trigger style
   - Shows color dot + project name when set, "Project" placeholder when empty
6. **Page wiring** (`apps/web/src/pages/user-management.tsx`):
   - Imported `useUpdateDefaultProject`, created `handleDefaultProjectChange` callback
   - Passed `onDefaultProjectChange` to column definitions

## Timetastic Department -> Default Project Data Migration

**Task:** Populate default projects from Timetastic department assignments.

**How it works:** TT users have a `departmentId` in their `raw_data` JSONB. TT departments have `id` and `name`. We mapped department names to Ternity project names:

| TT Department                 | Ternity Project    |
| ----------------------------- | ------------------ |
| Yosensi Team                  | Yosensi            |
| Yosensi Software Project Team | Yosensi            |
| DevOps Team                   | Devops - duty time |
| HR&Administration Team        | HR                 |
| Mercaso Project Team          | Mercaso            |
| Shaype Project Team           | Shaype             |
| Legal500                      | Legal500           |

**Gotcha:** There were 2 projects named "Legal500" — one active (client: Legal500), one inactive (client: External Corp). Initial UPDATE joined to both, causing some users to get the inactive one. Fixed with a follow-up UPDATE targeting the inactive project UUID.

**Result:** 25 users with TT accounts now have default projects. The remaining ~56 users with NULL are from Phase 2 user matching (deactivated TT users found only in absence records — no `stg_tt_users` rows, so no department info).
