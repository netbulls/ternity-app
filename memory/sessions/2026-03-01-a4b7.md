**Updated: `apps/web/src/pages/timer.tsx`**

- Added `selectedDate` state (defaults to today)
- Fetches full week's entries with `useEntries(weekFrom, weekTo)` — one API call for the whole week
- Derives selected day's data from the week fetch (no extra request)
- Week strip wired with `onSelectDate`, `onPrevWeek`, `onNextWeek`, `onToday`
- Keyboard navigation: Left/Right arrows change day, T returns to today

### Design decisions

- Single API call for the week (reuses existing `useEntries` hook with week range) — no new backend endpoints needed
- Week strip chips derive hours from `DayGroup.totalSeconds` — matches the V7 exploration design
- Timer bar hidden on non-today dates to avoid confusion (can't start a timer in the past)
- DayTimeline still renders for all dates (shows historical blocks)
- Keyboard shortcuts skip inputs/textareas to avoid conflicts with entry editing
- Daily target hardcoded to 8h (28800s) — can be made configurable later from working hours settings

## Fixes: timeline header + week strip calculations

User spotted two issues after testing the week navigation:

### 1. DayTimeline header hardcoded "Today's Timeline"

- Changed to contextual: "Today's Timeline" when viewing today, otherwise "Thursday's Timeline" etc.
- Also hid the NOW marker when viewing a non-today date (was showing the current time position on past/future day timelines, which is confusing)

### 2. WeekStrip chip hours wrong for cross-midnight entries

- The chips were using `DayGroup.totalSeconds` from the API, which is the entry's full duration — includes time from other days when an entry spans midnight
- Fixed by importing `calcTodaySeconds` from `day-timeline.tsx` and using it to compute day-clamped seconds for each day's chip
- This is the same clamping logic already used for the DayTimeline and the corrected DayGroup header on the timer page
