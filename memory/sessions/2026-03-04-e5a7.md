### Key Decision: Two Workforce Types, Contractor-First

#### The Legal Problem

In Poland, if contractors (B2B) go through a formal leave **approval** process, ZUS (social insurance authority) can treat this as evidence of an employment relationship — major legal risk. So contractors must NOT have an approval flow.

#### Decided Approach

1. **User profile gets `employment_type`** field — `contractor` vs `employee`
2. **Build contractor flow first** — simple "I'm booking time off" with no approval needed
3. **Add employee flow later** — formal request + manager approval (legally required for umowa o prace)
4. **For now, treat everyone as contractors** — even if someone is marked employee, approval flow doesn't exist yet

#### Status Model

- New status: **`autoconfirmed`** — used when contractor self-books time off (name makes it clear no human reviewed it)
- Existing `approved` status reserved for future employee flow (manager explicitly approves)
- DB enum `leave_status` needs to be extended: `pending`, `approved`, `rejected`, `cancelled` + **`autoconfirmed`**
- Contractor books time off -> immediately `autoconfirmed`
- Employee books time off -> `pending` -> manager approves -> `approved` (future)

### Booking Duration & Validation Decisions

1. **Partial-day bookings**: 1h increments up to 4h. Single date only. Aligns with presence 4-hour rule.
2. **Full-day bookings**: 4h+ = full day. Can span date ranges. Weekends + Polish public holidays auto-excluded from count.
3. **No overlapping bookings**: System rejects if dates conflict with existing active booking for same user.
4. **Public holidays**: Hardcode Polish holidays for 2025–2027 initially. Extend later.
