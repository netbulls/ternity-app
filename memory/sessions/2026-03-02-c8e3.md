## Context

User reported error on dev environment fuzzy search:

```json
{ "error": "function similarity(text, unknown) does not exist", "code": "42883" }
```

## Investigation

The `pg_trgm` extension is required for the `similarity()` function used in `GET /api/entries/search` endpoint (`apps/api/src/routes/entries.ts:394-470`).

Migration file `apps/api/drizzle/0015_pg_trgm_fuzzy_search.sql` exists but is **not registered** in the Drizzle journal — it was applied manually on local. Likely never applied on the dev database.

## Root Cause

The `pg_trgm` PostgreSQL extension was never enabled on the dev database. The migration `0015_pg_trgm_fuzzy_search.sql` was applied manually on local but skipped on dev.

## Fix

Need to run `CREATE EXTENSION IF NOT EXISTS pg_trgm;` and create the GIN index on the dev database.

## Prod — DONE

Applied on prod. Extension already existed (skipped), GIN index created successfully.

## Bug Fix: Timer Suggest clones entry instead of resuming

**Problem:** Selecting a past entry from the timer bar's suggest dropdown (TimerSuggest) creates a brand new entry (clone) instead of adding a new segment to the existing entry. The command palette (Cmd+K) correctly resumes by calling `POST /timer/resume/:id`.

**Root cause:** `handleSuggestSelect` in `timer-bar.tsx` was copying description/project/jira into form state and setting `lastStoppedEntryIdRef.current = null`. When user pressed Enter, `handleStart` fell through to `startTimer.mutate()` which creates a new entry via `POST /timer/start`.

**Fix:** Changed `handleSuggestSelect` to directly call `resumeTimer.mutate(entry.id)` — same as the command palette's `doEntryResume`. This adds a new segment to the existing entry instead of creating a clone. The suggest dropdown only appears when the timer is stopped, so no running-timer confirmation is needed.

**File changed:** `apps/web/src/components/timer/timer-bar.tsx` — `handleSuggestSelect` callback.

## Feature: Day-scoped entry display and timer

**Problem:** When resuming a task from yesterday today, the entry migrated entirely to today (grouped by `lastSegmentAt`). Duration showed ALL-TIME total across all days, not just the current day's work.

**User requirements:**

1. Entry should appear in EVERY day it has segments, not just the most recent day
2. Duration shown per entry should be day-scoped (only segments within that day)
3. Timer bar should show today-only elapsed time, not cumulative all-time total
4. Time range display should show day-clamped start/stop times

**Changes made:**

### Backend (`apps/api/src/routes/entries.ts`)

- Day grouping logic (step 6) now iterates each entry's segments and adds the entry to every day group where it has segments
- Multi-day segments (crossing midnight) correctly produce entries in both days
- Running segments use `Date.now()` to also appear in today's group

### Frontend — `EntryRow` (`apps/web/src/components/entries/entry-row.tsx`)

- Now accepts a `date` prop from `DayGroup`
- Computes `dayCompletedOffset` — sum of completed segment durations clamped to day boundaries
- Running segment's `startedAt` is clamped to day start for ticking
- Time range shows only segments overlapping the displayed day, with clamped start/stop times

### Frontend — `DayGroup` (`apps/web/src/components/entries/day-group.tsx`)

- Passes `group.date` to each `EntryRow`

### Frontend — `TimerBar` (`apps/web/src/components/timer/timer-bar.tsx`)

- Computes `todayCompletedOffset` — sum of completed segments clamped to today
- Running segment start clamped to today's midnight for ticking
- Timer display now shows only today's accumulated time

### Frontend — `DayTimeline` (`apps/web/src/components/timer/day-timeline.tsx`)

- Extracted `calcEntryDaySeconds()` utility for single-entry day-clamped duration
- Refactored `calcTodaySeconds()` to use `calcEntryDaySeconds()` internally
- No visual change — timeline already rendered day-clamped segments

**Build:** API and web both compile clean.
