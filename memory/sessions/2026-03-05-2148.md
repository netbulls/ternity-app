## UTC Timezone Bug Fix

**Problem:** Dashboard, entries, and stats APIs used UTC for all date calculations. For users in CET (UTC+1), this caused entries to appear on the wrong day — e.g., "Thursday" showing time that should be on "Wednesday" because the server's UTC clock had rolled past midnight while local time hadn't.

**Root cause:** All date grouping used `AT TIME ZONE 'UTC'` in SQL, `getUTCDay()`/`setUTCHours()` for boundaries, and `.toISOString().slice(0, 10)` for "today". For UTC+1, there's a 1-hour window each night where UTC date is one day ahead of local date.

**Fix:** Applied the same `ORG_TIMEZONE = 'Europe/Warsaw'` pattern that `team.ts` already used.

**Files changed:**

1. **`apps/api/src/routes/dashboard.ts`** — Full rewrite of date logic:
   - Import `ORG_TIMEZONE` from shared
   - Added helpers: `orgDateStr()`, `orgDateParts()`, `orgMidnightToUTC()`, `addDays()`, `dateDow()`
   - Week boundaries: computed from `orgDow` (org timezone day-of-week), not `getUTCDay()`
   - Month boundaries: computed from `orgYear`/`orgMonth`, not `getUTCFullYear()`/`getUTCMonth()`
   - SQL queries: `AT TIME ZONE ${ORG_TIMEZONE}` instead of `AT TIME ZONE 'UTC'`
   - Query boundaries: `orgMidnightToUTC()` for UTC timestamps that represent org timezone midnight
   - "Today" string: `orgDateStr(now)` instead of `now.toISOString().slice(0, 10)`
   - Heatmap days: iterate by org month, not UTC month
   - Working days left: iterate using `addDays()` from org timezone tomorrow

2. **`apps/api/src/routes/entries.ts`**:
   - Import `ORG_TIMEZONE` from shared
   - Added helpers: `orgMidnightToUTC()`, `addDays()`, `isoToOrgDate()`
   - `from`/`to` query params: converted via `orgMidnightToUTC()` instead of `T00:00:00.000Z` UTC
   - Default "today": `toLocaleDateString('en-CA', { timeZone: ORG_TIMEZONE })` instead of `toISOString().slice(0, 10)`
   - Day grouping: `isoToOrgDate(seg.startedAt)` instead of `seg.startedAt.slice(0, 10)` (UTC extraction)
   - Running segment "now": `orgTodayStr` instead of `new Date().toISOString().slice(0, 10)`
   - Adjustment segments: `isoToOrgDate(seg.createdAt)` instead of `.slice(0, 10)`

3. **`apps/api/src/routes/stats.ts`**:
   - Import `ORG_TIMEZONE` from shared
   - Added same `orgMidnightToUTC()` and `addDays()` helpers
   - Today/week boundaries computed in org timezone

4. **`apps/api/src/routes/leave.ts`**:
   - Import `ORG_TIMEZONE`
   - "Cannot book in the past" check uses org timezone for today

**Note:** `team.ts` was already correct — it was the reference implementation for this pattern.

**Runtime fix:** drizzle's `sql` template tag parameterizes `${ORG_TIMEZONE}` as `$1`, causing Postgres GROUP BY mismatch errors (`column must appear in GROUP BY`). Fixed by using `sql.raw("'Europe/Warsaw'")` via a `TZ_LITERAL` constant.
